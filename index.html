Project Structure

The Newman testing framework solely based on node js implementation which means we will have some configurations which are going to help run Newman through Jenkins.

1. Environment : It contains configuration files according to environment. i.e., dev, uat & production.

2. Report : It contains helper files which helps to generate after run Newman's status report.

3. Src : It contains postman collections under src\test\resources\newman.

4. Section 4 : These are the support files for nodejs implementation.

Environment Configuration

As stated in previous section environment configurations helps to setup project and helps use Newman as we do in Postman Runner.

The configuration primarily divided into three major environment i.e. dev, uat & production.

The configurations are written in JSON format. JSON stands for JavaScript Object Notation. It is an open standard file format, and data interchange format, that uses human-readable text to store and transmit data objects consisting of attribute–value pairs and array data types.

Before proceeding to next section part if you are not familiar with JSON, we suggest please have a quick tour about it and then proceed or if you know then lets jump to learning about configuration.

1. Environment variables

Here we get to know how we can set Environment Variables in Newman. Variables allow you to store and reuse values in your requests and scripts. By storing a value in a variable, you can reference it throughout your collections, environments, and requests—and if you need to update the value, you only have to change it in one place. Using variables increases your ability to work efficiently and minimizes the likelihood of error.

Postman supports the followingvariable scopes:

  • Global
  • Collection
  • Environment
  • Data
  • Local

1. enabled : If the provided variable should take participate in test or not.

2. key : The name of the key.

3. value : The value of key.

4. type: The data type of value. It can be text, boolean, numeric etc.

2. Execution Order

Execution order is an array which contains sequence of request which should get execute in given order. If any request needs to make special request from its response test, they can do it by usingPostman's Request Workflows.

Note: Request name must be unique in order to uniquely identify request.

3. Client Configuration

The config section or client configuration section will contains key-value configuration which helps to establish db connnection, providing dynamic data to requests and giving iteration data to help to test api with multiple data set.

1. Client Name: Here comes the name of the client.

2. Description: This defines about client and test and about its intention. Please make it elaborate so it should be understandable that what the test was for and what configuration with client is working.

3. Clean: This configuration is of type boolean which can only be true/false. True tells system to run test after cleaning defined database. This included re-creating new tables with updated seed data. If given false, table's re-creation and seed data part will gets skip out and test will run.

4. Seed Data Client Folder Name: It is the name of client folder defined in seed repository. This will help to run client specific seed data after default one.

5. Common Data: This section helps to define those configuration which could be common in each iteration. Instead of repeating it on iterations we can write it into this section. If same keys provided in iteration and common data, then iteration data will get replace by common one.

6. Iteration Data: These are set of data values that will take participate in collection run. Each iteration can have separate data values so that Api's can be tested with multiple data sets.

In this section we going to see how to use those dynamic variables that we just learned about in previous section.

You can use double curly braces to reference variables throughout the Postman user interface. For example, to reference a variable named "username" in your request auth settings, you could use the following syntax with double curly braces around the name:

{{username}}

When you run a request, Postman will resolve the variable and replace it with its current value.

For example, you could have a request URL referencing a variable as follows:

http://pricey-trilby.glitch.me/customer?id={{cust_id}}

Postman will send whatever value you currently have stored for the cust_id variable when the request runs. If cust_id is currently 3, the request will be sent to the following URL including query parameter:

http://pricey-trilby.glitch.me/customer?id=3

Alternatively, you could have a request body that accesses a variable by wrapping its reference in double-quotes:

{ "customer_id" : "{{cust_id}}" }

You can use variables in request URLs, parameters, headers, authorization, body, and header presets.
